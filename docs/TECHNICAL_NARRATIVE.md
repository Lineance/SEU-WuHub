# SEU-WuHub 校园信息智能问答平台技术报告

## 摘要

本文档详细介绍 SEU-WuHub 校园信息智能问答平台的技术架构与核心实现。该平台面向高校师生，整合教务处、图书馆等多部门通知公告，通过检索增强生成技术提供自然语言问答服务。系统采用前后端分离架构，前端基于 React 与 Next.js 构建单页应用，后端基于 FastAPI 构建 RESTful API 与 SSE 流式响应接口。数据存储采用 LanceDB 嵌入式向量数据库，实现结构化数据、全文索引与向量索引的统一管理。智能问答模块基于 ReAct 范式构建 Agent 系统，支持搜索、SQL 查询、网页抓取等多工具协同推理能力。

---

## 1. 引言

### 1.1 研究背景

高校校园信息分散于教务处、图书馆、学生处等多个部门官方网站，学生获取通知往往需要在多个站点之间切换。这种分散的信息架构增加了信息获取成本，也容易导致重要通知的遗漏。传统的站内搜索功能受限于单一数据源，无法跨部门检索语义相关的信息。

随着大语言模型技术的发展，基于检索增强生成的智能问答系统成为解决这一问题的有效方案。通过将用户问题转换为向量表征，在海量校园信息中进行语义检索，再由大语言模型生成符合上下文的回答，可以实现自然流畅的跨源信息问答体验。

### 1.2 项目目标

SEU-WuHub 项目旨在构建一个统一的校园信息聚合与智能问答平台，其核心目标包括以下几个方面。

首先是多源信息自动采集。系统需要支持配置化的爬虫方案，新增数据源仅需修改配置文件而无需改动代码，实现对教务处、图书馆等不同部门网站的信息采集。

其次是基于自然语言的智能问答。通过构建基于 ReAct 范式的 Agent 系统，使系统能够理解用户问题的意图，自动调用搜索工具获取相关信息，并生成准确流畅的回答。

第三是混合检索能力。系统需要同时支持基于向量的语义检索和基于关键词的全文检索，并通过融合算法综合两种检索方式的优势，提升检索结果的准确性和相关性。

第四是用户友好的界面交互。前端采用现代化的响应式设计，支持深色模式适配，提供流畅的 SSE 流式响应体验，让用户获得接近对话的交互感受。

### 1.3 设计约束

本项目在设计之初面临多重约束，这些约束直接影响了技术选型和架构设计。

交付周期限制为四周时间，这意味着必须采用单体优先的架构策略，拒绝微服务化带来的复杂度。部署方式限定为单机容器化，需要嵌入式存储方案以降低运维复杂度。预算限制要求云服务器成本控制在每年一千五百元以内，因此必须选择零外部服务依赖的技术栈。安全模型要求公网仅支持只读操作，写入操作通过本地 SSH 执行，防止外部恶意篡改数据。

## 2. 技术选型

### 2.1 前端技术栈

前端选用 React 19 作为 UI 框架，选择该版本的原因在于其修复了此前版本存在的服务端组件拒绝服务漏洞（CVE-2026-23864），该漏洞会影响 19.2.0 至 19.2.3 版本。

React 19 配合 Next.js 16 的 App Router 架构，实现了服务端组件与客户端组件的合理划分。服务端组件负责数据获取和静态渲染，客户端组件处理用户交互和动态更新。

UI 组件库选用 shadcn/ui，该组件库基于 Radix UI 的无障碍组件构建，提供一致的视觉风格和良好的可访问性支持。样式方案采用 Tailwind CSS 4，使用 oklch 色彩空间定义主题色，支持亮色和深色模式的自适应切换。

前端同时引入了 Framer Motion 实现页面过渡动画，Embla Carousel 实现轮播图组件，React Hook Form 配合 Zod 实现表单验证，以及 Recharts 实现数据可视化。

### 2.2 后端技术栈

后端选用 FastAPI 作为 Web 框架，选择原因包括其自动 OpenAPI 文档生成、原生异步支持，以及从 0.130.0 版本引入的 Rust-based Pydantic 序列化带来的性能提升。

数据存储是本项目的核心决策点。最终选择 LanceDB 作为统一存储引擎，原因在于其三个关键特性。首先是嵌入式部署，LanceDB 无需独立进程运行，数据存储在本地文件系统目录中，备份仅需复制目录。其次是多索引类型支持，LanceDB 同时支持向量索引（IVF-PQ）、全文索引（Tantivy）和结构化数据查询，无需组合使用多个数据库。第三是中文优化，Tantivy 内置对中文分词的支持，通过配置即可启用。

向量嵌入模型选用 BAAI/bge-large-zh-v1.5，该模型针对中文语义理解优化，输出 1024 维向量。标题嵌入则使用 paraphrase-multilingual-MiniLM-L12-v2，输出 384 维向量，平衡精度与性能。

大语言模型路由选用 LiteLLM，该库封装了 OpenAI、DeepSeek 等多个 LLM 提供商的 API，实现统一的调用接口，便于后续切换模型供应商。

爬虫框架选用 Crawl4AI，相比传统爬虫方案，Crawl4AI 内置对 JavaScript 渲染页面的支持，通过配置选择器规则即可适配不同结构的网站。

### 2.3 技术选型对比

为了清晰说明技术决策的合理性，以下从关键维度对比本项目选型与其他可行方案。

在向量数据库选型方面，相比 Pinecone、Weaviate 等云服务方案，LanceDB 的嵌入式特性使其无需额外的服务进程和费用。相比 Chroma、Qdrant 等开源方案，LanceDB 提供更完整的索引类型支持，且对中文全文检索的集成更为顺畅。

在爬虫框架选型方面，相比 Scrapy 等传统框架，Crawl4AI 对动态渲染页面的支持更为完善。相比 Playwright 配合自建爬虫的方案，Crawl4AI 提供了更高级的 API 和内置的 HTML 清洗功能。

## 3. 系统架构

### 3.1 整体架构

系统采用前后端分离架构，通过 Nginx 反向代理统一入口。Nginx 同时承担静态资源服务和 API 请求转发的职责，并通过配置限制外部请求仅支持 GET、HEAD、OPTIONS 方法，防止数据被外部恶意修改。

前端部署于 Nginx 容器中，以 Next.js 构建产出的静态文件形式提供服务。当用户访问根路径时，Nginx 将请求转发至 Next.js 容器获取页面资源。

后端部署于独立的 FastAPI 容器中，通过 Docker 网络与前端通信。API 网关层仅包含请求校验和响应格式化，不包含复杂业务逻辑。业务逻辑下沉至服务层，服务层负责协调 Agent 系统、检索引擎和数据访问层的工作。

数据层采用 LanceDB 作为统一存储引擎。一份数据同时承担结构化数据存储、全文索引和向量索引的职责，通过不同的查询接口满足差异化的检索需求。

爬虫模块独立于主系统运行，通过 CLI 接口触发执行。采集到的数据经过 ETL 管道处理后写入 LanceDB。

### 3.2 模块依赖关系

系统的后端模块按照依赖关系可划分为多个层次。

核心层位于依赖链的底部，向上提供配置管理和日志抽象等基础设施服务。该层不依赖任何其他模块，被所有上层模块使用。

数据层依赖于核心层，提供纯 IO 性质的数据访问能力。该层包含 LanceDB 连接管理、文章仓储和标签仓储，不包含任何业务逻辑判断。

检索层依赖于数据层，提供混合检索能力。该层封装了向量搜索、全文搜索和结果融合的具体实现细节，向上层屏蔽存储引擎的差异。

数据摄取层依赖于数据层，同时依赖嵌入模型服务。该层实现完整的 ETL 管道，包括数据验证、标准化、去重、向量化处理和标签匹配等功能。

Agent 层同时依赖检索层和大语言模型客户端，提供智能问答的核心推理能力。该层实现了 ReAct 范式的主循环逻辑，包括工具选择、观察结果处理和答案生成。

应用层位于依赖链顶端，仅负责组装协调工作。该层接收 HTTP 请求，调用相应服务层方法，将结果格式化后返回。

### 3.3 数据流向

系统的数据流向分为采集流程和查询流程两个主要方向。

采集流程从爬虫模块开始。爬虫根据 YAML 配置文件定义的规则抓取目标网页，通过 CSS 选择器提取正文内容和元数据。原始数据传入 ETL 管道后，首先经过验证器检查必填字段和格式，然后经过标准化处理将不同来源的数据格式统一。

去重处理在标准化之后执行。系统同时使用 URL 精确匹配和 SimHash 相似度匹配两种策略，确保新数据与已有数据不重复。

通过去重检查的数据进入向量化处理阶段。标题和正文分别通过对应的嵌入模型转换为向量表示。向量化完成后，系统计算文章内容与预定义标签的向量相似度，自动为文章打上相关标签。

最终处理完成的数据通过批量写入接口存入 LanceDB。写入过程使用事务保证原子性，避免因部分失败导致的数据不一致。

查询流程从用户提问开始。用户问题首先由 Agent 接收，Agent 根据问题内容判断需要调用的工具。如果问题涉及校园信息检索，Agent 调用搜索工具获取相关文档列表。搜索工具将问题转换为向量表征，在 LanceDB 中同时执行向量搜索和全文搜索，最后通过加权融合算法综合两种搜索的结果。

Agent 将检索到的文档作为上下文，结合对话历史调用大语言模型生成回答。回答通过 SSE 流式接口实时推送至前端，用户可以边接收边阅读。

## 4. 核心模块实现

### 4.1 Agent 系统

#### 4.1.1 ReAct 范式

Agent 系统的核心设计基于 ReAct 范式，即"推理与行动"交替进行的决策模式。该范式最早由普林斯顿大学研究者提出，核心思想是将大语言模型的推理能力与外部工具的执行能力相结合，使模型能够处理需要实时信息或执行计算的任务。

ReAct 范式的工作流程包含四个关键步骤。首先是思考阶段，模型分析当前状态和问题，决定下一步应该采取什么行动。其次是行动阶段，模型调用外部工具执行具体操作，如搜索信息或查询数据库。第三是观察阶段，模型获取工具执行的结果，将其作为下一轮推理的输入。第四是迭代，重复上述过程直到模型判断已收集足够信息生成最终答案。

这种范式的优势在于推理过程的透明性。每一轮思考和行动都被完整记录，便于追溯决策依据和排查问题。同时，工具执行结果的显式引入避免了纯语言模型的幻觉问题，确保回答基于真实检索到的信息。

#### 4.1.2 工具系统

工具系统是 Agent 与外部世界交互的桥梁。每个工具封装了特定的外部能力，通过标准化的接口供 Agent 调用。

搜索工具封装了检索引擎的能力。当 Agent 调用该工具时，系统将用户问题转换为向量表征，在 LanceDB 中执行混合搜索，返回与问题语义相关度最高的文档列表。搜索结果包含文档标题、摘要、来源和 URL 等关键信息。

SQL 工具封装了数据查询的能力。该工具接收结构化的过滤条件，返回符合条件的文档列表。相比搜索工具，SQL 工具更适合执行精确条件筛选，如按来源站点、发布日期或标签过滤。

抓取工具封装了网页内容获取的能力。当 Agent 需要获取文档原文时，调用该工具从指定 URL 抓取 HTML 内容。系统对抓取目标进行了域名限制，仅允许东南大学域名下的 URL，防止安全风险。

工具注册表负责管理所有可用工具的元信息，包括工具名称、描述和参数模式。当 Agent 需要选择工具时，注册表提供可用工具列表供大语言模型决策参考。

#### 4.1.3 对话记忆

对话记忆模块负责维护多轮对话的上下文信息。系统为每个用户会话维护独立的记忆存储，通过滑动窗口机制限制记忆长度，避免无限制增长。

记忆的内容包括用户问题和 Agent 回答的交替序列。每一轮对话结束时，当前的问答对被追加到对应会话的记忆存储中。当会话记忆超出滑动窗口范围时，早期的对话记录被丢弃，仅保留最近的若干轮对话。

这种设计的考虑在于，过长的对话历史会增加大语言模型的输入 token 消耗和推理延迟，同时早期对话对当前问题的参考价值往往有限。通过滑动窗口机制，系统在保持上下文连贯性的同时控制了资源消耗。

### 4.2 检索引擎

#### 4.2.1 混合检索架构

检索引擎的核心设计是混合检索架构，同时支持向量搜索和全文搜索两种检索模式，并根据配置权重融合两种模式的结果。

向量搜索基于语义相似度匹配。当用户输入查询时，系统首先将查询文本通过嵌入模型转换为向量表示，然后计算该向量与数据库中所有文档向量的余弦相似度，返回相似度最高的文档。向量搜索的优势在于能够捕捉语义相关性，即使查询词与文档用词不同，只要语义相近就能被检索到。

全文搜索基于关键词匹配。系统使用 Tantivy 全文搜索引擎，在文档的标题和正文字段上建立倒排索引。查询时，系统解析查询词的逻辑组合（AND、OR），在倒排索引中快速定位匹配的文档。全文搜索的优势在于精确匹配，对于包含明确关键词的查询，返回结果的相关性通常更高。

单一检索模式各有局限。向量搜索可能返回语义相关但关键词不匹配的结果，全文搜索则无法处理同义词和表达多样性。混合检索通过融合两种模式的结果，在精确性和泛化能力之间取得平衡。

#### 4.2.2 结果融合算法

融合算法的选择直接影响混合检索的效果。本系统采用 Reciprocal Rank Fusion 算法，该算法源自信息检索领域的研究成果，通过排名而非原始分数进行融合，具有良好的鲁棒性。

算法的核心思想是，如果一份文档在多种检索结果中都出现且排名靠前，则其综合相关性更高。具体计算时，每种检索模式根据文档排名赋予分数，排名越高分数越大，最终分数为各模式分数的加权和。

系统默认配置向量搜索权重为 0.6，全文搜索权重为 0.4。这个比例反映了系统设计者认为语义相似度对校园信息检索更重要，但关键词匹配也不可或缺的判断。实际应用中，可根据具体场景调整权重比例。

融合过程中还引入了位置衰减因子。排名靠后的文档获得的分数按指数衰减，避免排名差异过大时低排名文档获得过高权重。

#### 4.2.3 向量索引优化

向量索引是影响检索性能的关键因素。本系统采用 IVF-PQ 索引类型，这是一种结合了倒排文件和产品量化的复合索引类型。

倒排文件索引将向量空间划分为多个聚类，查询时首先定位到最相关的若干聚类，再在候选聚类内进行精确搜索。这种两阶段策略相比暴力穷举大幅减少了比较次数。

产品量化将高维向量分割为多个子向量，对每个子向量独立进行聚类编码。编码后的向量存储空间大幅压缩，同时保持可接受的检索精度。

索引创建时机也经过考量。系统仅在数据量达到一定规模（256 条记录）后创建向量索引。在此之前，暴力穷举的检索性能已经足够，且避免了小数据量下索引构建的资源浪费。

### 4.3 数据存储

#### 4.3.1 统一存储设计

本项目的一个核心设计决策是采用 LanceDB 作为统一存储引擎，同时承担结构化数据存储、全文索引和向量索引的职责。这一决策源于项目对部署简单性和运维成本的要求。

传统方案可能选择 PostgreSQL 配合 pgvector 扩展存储向量，Elasticsearch 存储全文索引。多个数据库的组合虽然能够满足功能需求，但带来了部署复杂、数据一致性和运维成本等一系列问题。

LanceDB 的嵌入式特性解决了这一困境。数据库以目录形式存储，备份仅需复制目录，无需执行数据库特定的导出命令。多个索引类型共存于同一数据文件中，通过不同的查询接口访问，不存在双写一致性问题。

这种设计也有其局限。LanceDB 作为嵌入式数据库，适合单机部署场景。对于需要横向扩展的分布式场景，该方案并不适用。这也是项目在当前阶段做出的合理权衡。

#### 4.3.2 数据模型

文章数据模型包含丰富的信息字段。核心字段包括新闻唯一标识、标题、原文链接、来源站点、作者信息。正文字段分别存储原始 Markdown 格式和纯文本格式，Markdown 格式用于前端渲染，纯文本用于全文索引和向量化。

时间字段存储发布日期和最后更新时间。版本字段记录爬取版本号，用于增量更新时判断文档是否发生变化。附件字段存储相关文件的 URL 列表，如通知公告的 PDF 附件。

向量字段存储标题嵌入和正文嵌入两组向量。分离存储支持差异化的权重配置，如标题匹配在最终结果中的权重可单独设置。标签字段存储系统自动匹配的标签 ID 列表。

标签数据模型相对简洁，核心字段包括标签唯一标识、显示名称、详细描述和分类。描述字段用于计算与文章内容的向量相似度，是标签自动匹配的关键输入。

#### 4.3.3 分页优化

文章列表的分页查询是一个容易被忽视的性能瓶颈。当数据量较大时，使用 OFFSET 分页会导致查询跳过大量数据，响应时间随页码增加而线性增长。

本系统采用预排序表的策略优化分页性能。具体做法是维护一张专门的排序表，存储文章 ID 与其全局排序位置和分类内排序位置的映射。当需要查询某一页时，首先在排序表中定位起始位置的文章 ID 列表，再通过 ID 批量获取文章详情。

这种两阶段查询的优势在于排序表的体积远小于主表，且只包含 ID 和排序位置两个整数字段，索引效率更高。批量获取详情时使用 IN 查询，由数据库进行优化。

### 4.4 标签系统

#### 4.4.1 自动标签匹配

标签系统是内容组织的重要辅助手段。系统支持根据文章内容自动匹配相关标签，减少人工标注的工作量。

匹配算法基于向量相似度计算。系统为每个预定义标签生成描述文本的向量表示。当新文章入库时，系统计算文章正文向量与所有标签向量的余弦相似度，选择相似度超过阈值的标签。

阈值设置考虑了准确率和召回率的平衡。较高阈值（如 0.75）确保匹配的标签高度相关，但可能遗漏部分相关标签。较低阈值（如 0.5）增加标签覆盖，但可能引入噪声标签。系统提供严格模式和宽松模式两种配置，供用户根据场景选择。

#### 4.4.2 标签配置管理

标签定义通过 YAML 配置文件管理，包含标签标识、显示名称、描述文本和分类。修改标签配置后，通过命令行工具重新初始化标签系统，即可使配置生效。

这种设计使得非技术人员也能通过编辑配置文件调整标签体系，无需修改代码和重新部署。标签分类（如学术讲座、考试通知、招聘信息等）的调整尤其适合这种管理模式。

## 5. 前端实现

### 5.1 应用结构

前端采用 Next.js App Router 架构组织页面和布局。App Router 是 Next.js 13 引入的新一代路由系统，通过文件系统约定自动生成路由映射，支持嵌套布局和并行路由等高级特性。

根布局包含全局 Provider 配置和基础 HTML 结构。ThemeProvider 包裹整个应用，提供深色模式支持。ReadingModeProvider 提供阅读模式的上下文切换，适合长时间阅读场景。

各功能页面通过嵌套布局共享通用的页面框架。AppShell 组件封装了侧边栏、顶栏和 AI 助手等通用 UI 元素，各页面嵌套在该布局内部，避免重复代码。

### 5.2 组件设计

组件设计遵循原子设计方法论，将界面元素划分为原子、分子、有机体等层次。原子组件如按钮、输入框等提供最基础的 UI 能力。分子组件由多个原子组合而成，实现具体的 UI 功能。有机体则是完整的页面区块。

AI 助手组件是前端的核心交互组件，采用会话式界面设计。用户输入问题后，系统通过 SSE 流式接收 Agent 的响应内容，实现打字机效果。响应过程中，系统根据事件类型（思考中、工具调用、答案生成）展示不同的 UI 状态，让用户了解 Agent 的工作进展。

文章卡片组件展示文章摘要信息，包含标题、来源、发布日期和标签。卡片支持点击跳转详情页和收藏操作。收藏数据存储于浏览器 localStorage，跨会话持久化。

### 5.3 状态管理

前端状态管理根据数据的特性和使用范围选择不同策略。

用户偏好设置（如主题模式、阅读模式）通过 React Context API 管理。这些数据需要在组件树的多处被访问，但更新频率较低，适合使用 Context 的发布-订阅模式。

搜索参数和筛选条件通过 URL 参数管理。这种设计使得分享搜索结果成为可能，同时利用浏览器的后退功能实现筛选历史的回退。

对话会话和收藏数据存储于 localStorage。localStorage 的优势在于跨标签页共享和浏览器关闭后持久化，适合用户生成内容类的数据存储。

### 5.4 SSE 流式通信

SSE（Server-Sent Events）用于实现服务端向客户端的单向流式数据传输。相比 WebSocket，SSE 的协议更轻量，且基于 HTTP 协议便于穿越防火墙和代理。

流式聊天的实现流程如下。用户提交问题后，前端发起 POST 请求并指定 Accept: text/event-stream 头部。服务端保持请求连接，通过 EventSourceResponse 持续发送事件数据。事件数据包含事件类型和数据载荷，格式为 event: type\\ndata: payload\\n\\n。

前端通过 ReadableStream API 读取响应内容。StreamReader 逐块读取数据，通过 TextDecoder 解码后按分隔符切分消息。每条消息解析后，根据事件类型触发相应的 UI 更新。

SSE 连接的断开由服务端控制。当 Agent 完成回答或达到最大步数限制时，服务端发送空消息关闭连接。前端检测到连接关闭后，停止读取循环，完成整个交互流程。

## 6. 部署与运维

### 6.1 容器化部署

系统通过 Docker Compose 实现容器化部署，包含后端、前端两个服务容器和 LanceDB 数据卷。

后端容器运行 FastAPI 应用进程，通过 uvicorn 作为 ASGI 服务器。容器启动时执行依赖安装和数据库初始化，健康检查接口返回服务状态和版本信息。

前端容器运行 Next.js 构建产物，通过 Node.js 的 HTTP 服务器提供静态文件服务。相比开发模式的生产构建，静态文件服务减少了不必要的运行时开销。

Nginx 作为反向代理部署在宿主机上（也可容器化），统一接收外部请求并转发至相应服务。Nginx 同时承担静态资源缓存和请求限流的职责。

### 6.2 数据备份

由于采用嵌入式存储，数据备份极为简单。只需复制 LanceDB 数据目录即可完成完整备份，恢复时将备份目录还原至原位置。

备份策略建议采用定时任务执行增量备份。考虑到校园信息的更新频率，每日一次备份已足够满足需求。备份文件可上传至对象存储服务，实现异地容灾。

### 6.3 监控告警

系统目前通过健康检查接口提供基本的可用性监控。健康检查返回数据库连接状态和索引状态，可接入外部监控系统实现告警通知。

未来可扩展的监控指标包括 API 响应延迟、检索结果质量分数、标签匹配覆盖率等。这些指标有助于评估系统性能和优化方向。

## 7. 结论

本文档详细介绍了 SEU-WuHub 校园信息智能问答平台的技术架构与核心实现。系统通过前后端分离架构、ReAct 范式 Agent、混合检索引擎和嵌入式向量数据库等关键技术，实现了多源信息聚合与自然语言问答的核心功能。

项目的技术选型始终围绕交付周期、部署成本和安全约束等现实因素展开，在功能完整性和实现复杂度之间取得了合理平衡。LanceDB 的引入简化了存储架构，ReAct 范式的采用提升了问答系统的可解释性，混合检索设计则在精确性和泛化能力之间找到了合适的平衡点。

系统的模块化设计使得各组件职责清晰，便于后续的功能扩展和性能优化。主要的优化方向包括：引入缓存层提升热门查询的响应速度、增加用户反馈机制迭代检索质量、以及支持更多数据源扩大信息覆盖范围。
